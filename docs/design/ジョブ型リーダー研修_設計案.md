# ジョブ型リーダー研修プログラム 設計案

作成日: 2026-06-15

## 1. 目的・背景
- 従来の「経験年数で昇格 → OJTで何となく学ぶ」メンバーシップ型から、**「リーダー職の職務（ジョブ）を明文化し、それを遂行できる人材を育成・評価する」ジョブ型**へ転換する。
- 育成のゴール＝**職務定義書（ジョブディスクリプション）に書かれた行動を、独力で遂行できる状態**。

## 2. リーダー職の職務定義（業務報告から抽出）
日々の業務報告タスクを「職務」として再定義。

| # | 報告タスク | リーダー職務（ジョブ） |
|---|-----------|----------------------|
| 1 | 諸々レビュー | 成果物の品質保証（設計・コード・ドキュメントのレビュー） |
| 2 | 諸々調整 | 社内外・顧客との折衝・調整 |
| 3 | 諸々フォロー | メンバー育成と進捗マネジメント |
| 4 | SES本番適用作業 | リリース管理・本番適用の意思決定とリスク管理 |
| 5 | パッケージシステム不具合改修 | 障害対応・原因分析・恒久対策 |

## 3. スキルマップ（職務 → 必要コンピテンシー）

| 職務 | 必要スキル | レベル定義（到達目標） |
|------|-----------|----------------------|
| ① 品質保証（レビュー） | レビュー観点設計／指摘の言語化／品質基準の策定 | レビュー指摘を「個人の好み」でなく「基準」で説明できる |
| ② 調整・折衝 | 利害調整／合意形成／顧客折衝／エスカレーション判断 | 対立する要求を整理し、落としどころを提示できる |
| ③ 育成・進捗管理 | 1on1／タスク分解／進捗の可視化／遅延の早期検知 | メンバーの躓きを遅延化する前に検知・支援できる |
| ④ リリース管理 | 本番適用手順設計／リスク評価／Go/NoGo判断／切り戻し計画 | 本番作業のリスクを洗い出し、判断根拠を説明できる |
| ⑤ 障害対応 | 原因分析（なぜなぜ）／影響範囲特定／恒久対策／再発防止 | 場当たり対応でなく恒久対策まで設計できる |

## 4. カリキュラム（モジュール構成）

| モジュール | 内容 | 形式 | 素材（実タスク連動） |
|-----------|------|------|--------------------|
| M0 イントロ | ジョブ型の考え方・自職務の理解 | 講義＋ワーク | 職務定義書を自分で記述 |
| M1 レビュー力 | レビュー観点・指摘の伝え方 | ケース演習 | 「諸々レビュー」の実例題材 |
| M2 調整・折衝力 | 合意形成・顧客折衝ロールプレイ | ロールプレイ | 「諸々調整」の実ケース |
| M3 マネジメント | 進捗管理・1on1・フォロー手法 | 演習＋実践 | 「諸々フォロー」の振り返り |
| M4 リリース管理 | 本番適用のリスク管理・判断 | ケース演習 | 「SES本番適用作業」を教材化 |
| M5 障害対応 | 原因分析・恒久対策立案 | 障害演習 | 「不具合改修」を題材に再発防止策 |

> ジョブ型のポイント：**各モジュールに「到達基準（できる/できない）」を設定**し、受講者は基準を満たすと"認定"される。研修＝知識インプットでなく、職務遂行能力の認定プロセスとする。

## 5. 評価・到達基準
- 各モジュールに**チェックリスト形式の到達基準**を設定（例：M5＝「なぜなぜ分析を5段掘り下げ、恒久対策を提示できた」）。
- 最終評価は**実タスクでの行動評価（OJT連動）**で実施。研修内の演習だけで完結させない。
- 評価結果を職務定義書のレベルにマッピングし、昇格・処遇に接続。

## 6. 進め方（スケジュール案）
1. **職務定義書の確定**（誰に・何を求めるか合意）← 最優先
2. スキルマップ・到達基準の作成
3. 実タスクからの教材化（M1〜M5）
4. パイロット実施 → フィードバック → 本展開

## 7. 次のステップ（深掘り候補）
- **A.** リーダー職の「職務定義書（ジョブディスクリプション）」を1枚にまとめる
- **B.** 各モジュールの「到達基準チェックリスト」を具体的に作り込む
- **C.** 報告タスクの実ケースを使った演習問題（M1〜M5）を作る
