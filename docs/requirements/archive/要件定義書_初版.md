# リーダー研修 評価システム 要件定義書（一次案）

> 本書は打合せメモを元にしたたたき台です。確定事項ではなく、議論のための叩き台として記述しています。

## 1. プロジェクト概要

### 1.1 背景・目的
- リーダー研修の成果物として「ジョブ型エンジニア」評価の仕組みを構築する
- 人事考課委員会への提出資料を作成しやすくする
- 部の評価軸から脱却できる汎用的な評価の仕組みを提示する
- メンバーの伸びしろを可視化する
- 案・仕組みが評価されればプロダクト化の余地あり（スモールスタート前提）

### 1.2 対象範囲（スコープ）
- **評価対象等級**: Rクラスを主対象（S/Mへの早期昇格促進が会社メリットとなる想定）
  - S/M巻き込みは難易度高のため一次対象外
- **対象部署（8月目標）**: PTUを最優先、可能であればSRU、Rエンジニア、VUC（協力度次第）
  - 現実的には2部署程度
- **対象職種**: 開発 / 営業（既存の2分類を踏襲）

### 1.3 体制
| 領域 | 担当 |
|---|---|
| 機能要件 | 本荘さん・幡野さん |
| 非機能要件 | 折登さん・野原さん |
| AI活用（実装側） | 野原さん・折登さん |
| 評価・検証 | その他メンバー |

### 1.4 マイルストーン
- 8月: 一部部署でのスモールスタート稼働
- 4Q: 人事考課委員会への提出に合わせた本運用
- 期日は人事考課委員会のスケジュールに合わせる（日付・プロセス要確認）

## 2. 技術スタック

| 区分 | 採用技術 |
|---|---|
| フロントエンド | Next.js (App Router想定) |
| スタイリング | Tailwind CSS |
| バックエンド | **TypeScript (Node.js)** をサーバレス関数（FaaS）上で実行する REST API |
| API形式 | REST API (JSON) |
| 認証 | M365（Microsoft Entra ID）連携を第一候補 / 手動ID登録をフォールバック |
| インフラ | **クラウド・サーバレス前提**（従量課金・scale-to-zero でコスト最小化） |
| ソース管理 | Git |
| IaC | Terraform 等でインフラをコード管理（環境再現性・ポータビリティ確保） |

### 2.1 アーキテクチャ方針
- フロント／バックを分離した SPA + REST 構成
- **サーバレス（FaaS + マネージドDB）を基本**とし、常時起動サーバを持たない＝アイドルコストをゼロに近づける
- 利用が評価期間に集中するワークロード特性に対し、**使った分だけ課金**される構成が最もコスト効率が高い
- インフラ依存をコードに持ち込まず、IaC + Git で環境を再現・移植可能に保つ
- バックエンドのランタイムは **TypeScript (Node.js)** に決定。フロント(Next.js)と言語統一し、FaaS のコールドスタートを軽量化（詳細 3.10）

## 3. 非機能要件

> **前提**: コストを最優先に抑えるため、クラウド上のサーバレス構成で構築する。
> 利用が評価期間に集中し平常時はほぼアイドルというワークロード特性（同時接続10以下）に対し、
> 「常時起動サーバを持たず、使った分だけ課金（従量課金・scale-to-zero）」を基本方針とする。

### 3.0 品質特性の優先順位

| 優先 | 特性 | 理由 |
|:-:|---|---|
| ★★★ | コスト効率 | スモールスタート・社内利用。ランニング月額を最小化することが最大要件 |
| ★★★ | セキュリティ／機密性 | 評価情報は機微情報。漏洩は人事的に致命的 |
| ★★☆ | 拡張性 | 将来 R100（500名・全社展開）／プロダクト化を見据え、作り直さず伸ばせること |
| ★★☆ | 保守性・ポータビリティ | 少人数運用。特定クラウドへのロックインを避け移植可能に |
| ★☆☆ | 性能・可用性 | 小規模・期間集中型のため過剰品質は不要。SLAは緩めで可 |

### 3.1 インフラ・ホスティング（サーバレス）

- **構成方針**: マネージドサービスのみで構成し、サーバの構築・運用・パッチ当てを持たない
  - フロント: 静的ホスティング＋CDN（Next.js を静的/SSR配信）
  - API: FaaS（関数）。リクエスト時のみ起動し、無リクエスト時は課金ゼロ
  - DB: サーバレス／従量課金のマネージドDB（scale-to-zero または最小構成）
  - ファイル（PDF/Excel出力等）: オブジェクトストレージ
- **クラウド選定方針**: 既存IT基盤が **M365（Teams / SharePoint / Entra ID）** 中心であることから、
  認証・運用の親和性が高い **Azure を第一候補**、AWS を対抗候補とする（GCPも可）。
  最終選定は 3.10 のコスト試算と無料枠で判断。**要確認事項**（後述）。

  | 役割 | Azure（第一候補） | AWS（対抗） | 備考 |
  |---|---|---|---|
  | フロント配信 | Static Web Apps | S3 + CloudFront / Amplify | CDN込み |
  | API（FaaS） | Functions（従量課金プラン） | Lambda | scale-to-zero |
  | DB | Cosmos DB サーバレス / SQL DB サーバレス | DynamoDB / Aurora Serverless v2 | 従量・自動停止 |
  | 認証 | Entra ID (M365流用) | Cognito + Entra ID 連携 | 後述 3.6 |
  | ストレージ | Blob Storage | S3 | 出力ファイル等 |
  | 監視 | Application Insights | CloudWatch | ログ・メトリクス |

- **CDN/WAF**: クラウド標準のCDN＋WAFを利用（Cloudflare 無料枠の併用も可）

### 3.2 認証・認可

- **認証**: M365（Microsoft Entra ID）による SSO を第一候補。社員アカウントを流用しID管理コストとパスワード管理リスクを削減
  - Entra ID が使えない場合のフォールバックとして手動ID登録＋クラウドの認証基盤（Cognito 等）
- **認可（権限モデル）**: グループベースのロール管理（RBAC）
  - 一般メンバー（自己評価・他者評価）
  - 評価者（上長・部次長）
  - 人事考課委員会メンバー（閲覧）
  - 管理者（マスタ・項目管理）
- 360°評価は匿名集約のため、**入力者本人を集計結果から逆引きできない**データ設計とする（設計案の「全て匿名」要件に対応）

### 3.3 性能

- 同時利用想定: 対象2部署 × 数十名 → **同時接続10以下**。評価期間（クォーター末等）にアクセスが集中
- 画面応答: 通常操作で 1〜2 秒以内を目標（FaaS の**コールドスタートを許容範囲に収める**ことが論点 → 3.10）
- バッチ的なスコア集計・帳票出力は数十秒程度を許容

### 3.4 可用性

- 社内利用・期間集中型のため **SLA は緩め**（平日日中の利用を想定、99% 程度で可）
- マネージドサービスの標準冗長性に委ね、独自の冗長化・マルチリージョンは行わない（コスト優先）
- 計画停止・障害時はメンテナンス告知＋期日調整で運用回避

### 3.5 拡張性・スケーラビリティ

- サーバレスのオートスケールにより、**全社展開（〜500名規模／R100）でもインフラ構成を変えずスケール**可能
- スモールスタート（2部署）→ 全社 → 外販プロダクト化、の各段階で**作り直さない**ことを設計上の前提とする
- マルチテナント化（プロダクト化時）を見据え、データにテナント/組織キーを持てる構造を初期から意識

### 3.6 セキュリティ

- 評価情報は**機微情報**として扱う（最小権限・必要範囲のみ閲覧）
- 通信は全て **HTTPS（TLS）** 化、保存データは**暗号化（クラウド標準の保存時暗号化）**
- アクセス制御: 当面は **社内からのアクセスに限定**（Entra ID 条件付きアクセス／IP制限／VPN等のいずれかで実現。社内ネットワーク前提を維持）
- **監査ログ**: 評価入力・閲覧・エクスポート操作を記録（誰が・いつ・何を）
- シークレット（接続文字列・APIキー）はクラウドのシークレット管理（Key Vault / Secrets Manager）で管理し、コード・Gitに含めない
- 脆弱性対応: マネージドサービス採用によりOS/ミドルのパッチ責任をクラウド側に移譲

### 3.7 データ・バックアップ

- **履歴保存**: クォーター毎 or 年次で評価履歴を保持（経年の伸びしろ可視化に必要）
- **バックアップ**: マネージドDBの自動バックアップ（PITR等）を利用。追加コストを抑えつつ復旧可能性を担保
- **データ所在**: 国内リージョンを選択（人事情報のため）
- インポート機能は一次案では非対応（手動登録）。エクスポート（PDF/Excel）は機能要件側で対応

### 3.8 運用・監視

- 少人数運用前提。**運用作業ゼロに近いマネージド構成**を選ぶ
- ログ・メトリクスはクラウド標準（Application Insights / CloudWatch）に集約
- デプロイは Git 連携の CI/CD（GitHub Actions 等）で自動化、IaC で環境を再現
- コスト監視: 予算アラート（バジェット）を設定し、想定超過を早期検知

### 3.9 保守性・ポータビリティ

- インフラ依存をコードに持ち込まず、IaC＋Git で**他クラウドへの移植余地**を残す（ロックイン回避）
- フロント／バック分離、REST 契約の明確化により、各層を独立して改修可能に

### 3.10 コスト

- **目標**: スモールスタート期のランニングコストを **月額数千円〜（無料枠内に収まれば実質ゼロ）** に抑える
- サーバレスの従量課金＋各クラウドの**無料利用枠**（関数の月間無料実行回数、DB最小構成等）を最大活用
- コストドライバと抑制策:
  | ドライバ | 抑制策 |
  |---|---|
  | 常時起動サーバ | 持たない（FaaS の scale-to-zero） |
  | DB のアイドル課金 | サーバレスDB／自動一時停止／最小スループット |
  | データ転送・CDN | 無料枠内に収める、出力ファイルは保持期間を設定 |
  | コールドスタート対策の常時暖機 | 小規模では基本不要。必要時のみ最小限 |
- **バックエンドランタイム（決定済み）**: **TypeScript (Node.js)** を採用
  - 当初案の Java(Spring Boot) は FaaS でコールドスタートが重くコスト/応答に不利なため見送り
  - フロント(Next.js)と言語を統一 → 学習・保守コスト削減、コールドスタート軽量化の両得
- クラウド利用料の**コスト負担部署**は引き続き要確認（プロダクト化の収益で回収する前提も含め整理）

## 4. 機能要件

### 4.1 業務フロー
```
[1] 評価項目設定（管理者・部次長）
        ↓
[2] 自己評価入力（被評価者）
        ↓
[3] 360°評価入力（上長・同僚・メンバー）
        ↓
[4] サマリー集計・スコア算出
        ↓
[5] ダッシュボード閲覧（余裕があれば実装）
        ↓
[6] 人事考課委員会向け出力
```

### 4.2 機能一覧（スコープは後ほど確定）

#### A. 認証・ユーザー管理（余裕があれば）
- A-01 ログイン（手動登録IDによる）
- A-02 ユーザー登録（管理者操作）
- A-03 グループ・部署マスタ管理
- A-04 評価者・被評価者の関係定義
- A-05 権限ロール割当

#### B. 評価項目管理（余裕があれば）
- B-01 評価項目マスタ管理（部次長が項目を捻出する想定）
- B-02 職種別（開発/営業）項目テンプレート
- B-03 評価軸カテゴリ管理
  - 資格
  - 技術力
  - リーダーシップ
  - その他（伸びしろ系項目）
- B-04 性格診断系（MBTI的）の選択式項目設定
- B-05 評価期（クォーター/期日）設定

#### C. 自己評価（必須）
- C-01 自己評価入力フォーム（選択式中心で省力化）
- C-02 入力ステータス管理（未着手／入力中／提出済）
- C-03 期日通知
- C-04 過去自己評価の参照

#### D. 360°評価（必須）
- D-01 評価者割当（上長・同僚・メンバー）
  - 評価者の範囲をどこまで含めるかは要決定（メンバークラスを含めるか）
- D-02 匿名／記名の切替設定
- D-03 評価入力フォーム
- D-04 入力ステータス管理
- D-05 期日通知・リマインド

#### E. 集計・サマリー（必須）
- E-01 評価スコア集計
  - 上長：メンバー比重設定
  - 課題達成スコア／メンバー視点スコアの分離
  - メンバー人数差の正規化処理
- E-02 絶対評価／相対評価の切替（要決定）
- E-03 サマリーレポート出力
- E-04 人事考課委員会向けエクスポート（PDF/Excel）

#### F. ダッシュボード（余裕があれば）
- F-01 個人ダッシュボード（自己評価×他者評価のギャップ可視化）
- F-02 部署ダッシュボード（部次長向け）
- F-03 伸びしろ・成長トレンド可視化

#### G. 管理機能（余裕があれば）
- G-01 評価サイクル管理
- G-02 ステータス一覧・滞留検知
- G-03 マスタ管理（部署・職種・等級）

### 4.3 評価ロジック検討事項（要決定）
- 上長スコアとメンバースコアの**比重**
- 最終スコアの定義
  - 課題達成度に対するスコア
  - メンバーから見た評価スコア
  - 両者の合成方法
- メンバー人数が現場により異なるため、**人数正規化**のロジック
- **絶対評価／相対評価**の選択
- SmartHR等の既存SaaSとの差別化軸
  - SmartHR: 部・課の成果重視
  - 本システム: 組織非依存・フラットな汎用評価

## 7. REST API 項目（一次案・抜粋）

> 詳細スキーマは別途定義。エンドポイント命名と責務分離の方針のみ提示。

### 7.1 認証
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET  /api/auth/me`

### 7.2 ユーザー
- `GET    /api/users`
- `POST   /api/users`
- `GET    /api/users/{id}`
- `PUT    /api/users/{id}`
- `DELETE /api/users/{id}`

### 7.3 評価項目マスタ
- `GET    /api/eval-items`
- `POST   /api/eval-items`
- `PUT    /api/eval-items/{id}`
- `DELETE /api/eval-items/{id}`
- `GET    /api/eval-items/templates?jobType={dev|sales}`

### 7.4 評価サイクル
- `GET  /api/eval-cycles`
- `POST /api/eval-cycles`
- `GET  /api/eval-cycles/{id}`
- `PUT  /api/eval-cycles/{id}/status`

### 7.5 自己評価
- `GET  /api/self-evaluations?cycleId={id}&userId={id}`
- `POST /api/self-evaluations`
- `PUT  /api/self-evaluations/{id}`
- `POST /api/self-evaluations/{id}/submit`

### 7.6 360°評価
- `GET  /api/peer-evaluations?cycleId={id}&targetUserId={id}`
- `POST /api/peer-evaluations`
- `PUT  /api/peer-evaluations/{id}`
- `POST /api/peer-evaluations/{id}/submit`
- `GET  /api/peer-evaluations/assignees?cycleId={id}&userId={id}`

### 7.7 集計・サマリー
- `GET /api/summaries/{cycleId}/{userId}`
- `GET /api/summaries/{cycleId}/department/{deptId}`
- `GET /api/summaries/{cycleId}/export?format={pdf|xlsx}`

### 7.8 ダッシュボード（任意）
- `GET /api/dashboards/personal/{userId}`
- `GET /api/dashboards/department/{deptId}`

### 7.9 マスタ
- `GET /api/masters/departments`
- `GET /api/masters/job-types`
- `GET /api/masters/grades`
- `GET /api/masters/roles`
