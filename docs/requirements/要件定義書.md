## 1. プロジェクト概要

### 1.1 背景・目的
- リーダー研修の成果物として「ジョブ型エンジニア」評価の仕組みを構築する
- 人事考課委員会への提出資料を作成しやすくする
- 部の評価軸から脱却できる汎用的な評価の仕組みを提示する
- メンバーの伸びしろを可視化する
- 案・仕組みが評価されれば全社展開の余地あり（スモールスタート前提）。外販プロダクト化は行わない

### 1.2 対象範囲（スコープ）
- **評価対象等級**: Rクラスを主対象（S/Mへの早期昇格促進が会社メリットとなる想定）
  - S/M巻き込みは難易度高のため一次対象外
- **対象部署（8月目標）**: PTUを最優先、可能であればSRU、Rエンジニア、VUC（協力度次第）
  - 現実的には2部署程度
- **対象職種**: 開発 / 営業（既存の2分類を踏襲）

### 1.3 体制
| 領域 | 担当 |
|---|---|
| 機能要件 | 本荘さん・幡野さん |
| 非機能要件 | 折登さん・野原さん |
| AI活用（実装側） | 野原さん・折登さん |
| 評価・検証 | その他メンバー |

### 1.4 マイルストーン
- 8月: 一部部署でのスモールスタート稼働
- 4Q: 人事考課委員会への提出に合わせた本運用
- 期日は人事考課委員会のスケジュールに合わせる（日付・プロセス要確認）

## 2. 技術スタック

| 区分 | 採用技術 |
|---|---|
| アプリ（フロント＋API） | **Next.js (App Router / SSR) フルスタック** — UI と API を**単一アプリ**で提供（Docker コンテナで常駐）。独立したバックエンドサービスは持たない |
| スタイリング | Tailwind CSS |
| バックエンド (API) | Next.js の **route handlers / server actions**（`app/api/...`）で実装。ランタイムは Node.js / TypeScript |
| API形式 | REST (JSON)。Next.js route handlers で提供 |
| AI / LLM | **Claude（Anthropic API）**。既定モデルは最新・最上位の **Claude Opus 4.8（`claude-opus-4-8`）**。用途により高速・低コストの Sonnet / Haiku も選択。公式 SDK は **TypeScript（`@anthropic-ai/sdk`）** で UI/API と言語統一 |
| 認証 | M365（Microsoft Entra ID）連携を第一候補 / 手動ID登録をフォールバック |
| インフラ | **社内IDC（オンプレミス）前提**。物理マシン（Windows Server）上の **Hyper-V** に **Ubuntu（最新LTS）VM** を構築し、**Docker** でアプリを運用 |
| コンテナ管理 / PaaS | **Coolify**（OSS のセルフホスティング PaaS）。リバースプロキシ + Docker管理 + GitHub連携 + CI/CD をセットで提供 |
| 外部公開 | **Cloudflare Tunnel**（特定ポートのみ Cloudflare 経由で安全に公開） |
| ソース管理 / CI/CD | **GitHub**。`main` / `dev` ブランチからそれぞれ CI/CD で自動デプロイ |
| ドメイン | `*.comthink.co.jp` または `*.dev-nct.jp`（社内ですぐ提供可能） |
| ポータビリティ | **Docker化**により特定環境へのロックインを回避し移植性を確保（AWS等クラウドへの移行は現状検討外） |

### 2.1 アーキテクチャ方針
- **Next.js (App Router) のフルスタック構成**: UI（SSR）と API（route handlers / server actions）を**単一アプリ**で提供する。独立したバックエンドサービスは持たない。**Docker コンテナとして常駐**させる
- スモールスタート期は**社内IDCの既存物理マシン上の VM 1台**に集約し、**追加のランニングコストを実質ゼロ**にすることを最優先とする
- アプリはすべて Docker 化し、**Coolify** で一元管理（デプロイ・リバースプロキシ・ログ・GitHub連携を集約）。インフラ個別構築の手間を最小化する
- **作り直さない方針**: Docker を共通の実行単位とすることで、スモールスタート（IDC）→ 全社展開（IDC の VM リソース増強で対応）の各段階で**アプリ本体を作り直さずに済む**構成とする
- ランタイムは Next.js が動作する **Node.js / TypeScript** に統一。UI と API を同一言語・同一アプリで扱い、学習・保守コストを削減
- コンテナ常駐構成のため、サーバレス特有の**コールドスタートは発生しない**（応答が安定）
- **AI活用は Claude（Anthropic API）を前提**とする。想定用途は、評価コメントの要約・整形、自己評価／360°評価のサマリー生成、人事考課委員会向け資料のドラフト生成、伸びしろ・成長トレンドの分析コメント等。Next.js のサーバ側（route handlers / server actions）から公式 SDK 経由で呼び出す。機微情報の取り扱いは 3.6 / 3.7 に従う

## 3. 非機能要件

> **前提**: スモールスタート期は **社内IDC（本社近傍のデータセンターのラックに設置済みの物理マシン）** のリソースを活用する。
> 既に稼働している Windows Server 上に **Hyper-V で Ubuntu VM** を建て、**Docker + Coolify** でアプリを運用する。
> 既存ハードウェアを使うため**追加のランニングコストは実質ゼロ**。利用が評価期間に集中し平常時はほぼアイドル（同時接続10以下）というワークロード特性に対し、十分なスペックを確保できる。
> 全社展開で規模拡大が必要になった段階は、まず **IDC の VM リソース増強**で対応する（AWS等クラウドへの移行は現状検討外。当初検討の記録は付録A）。

### 3.0 品質特性の優先順位

| 優先 | 特性 | 理由 |
|:-:|---|---|
| ★★★ | コスト効率 | スモールスタート・社内利用。**既存IDCハードウェアの活用で追加ランニングコストを実質ゼロ**に抑えることが最大要件 |
| ★★★ | セキュリティ／機密性 | 評価情報は機微情報。漏洩は人事的に致命的 |
| ★★☆ | 拡張性・ポータビリティ | 将来 R100（500名・全社展開）を見据え、**Docker化で作り直さず移植・拡張**できること |
| ★★☆ | 保守性 | 少人数運用。Coolify による一元管理で運用負荷を下げる |
| ★☆☆ | 性能・可用性 | 小規模・期間集中型のため過剰品質は不要。SLAは緩めで可（オンプレ単一構成のため可用性はトレードオフ） |

### 3.1 インフラ・ホスティング（社内IDC / オンプレミス）

- **設置場所**: 本社近傍のデータセンター（社内で「IDC」と呼称）のラックに設置済みの物理マシンを利用。新規調達不要。
- **基盤**: 物理マシン（Windows Server、複数台稼働中）の **Hyper-V** 上に **Ubuntu（最新LTS）VM** を構築。
- **VM割当スペック（割当可能な想定）**:

  | リソース | 割当 | 備考 |
  |---|---|---|
  | CPU | 4 Core | 大抵のアプリはこれで満足に動作 |
  | RAM | 16GB〜32GB | まずは様子見。当面この想定で開始し、必要に応じ増減 |
  | ストレージ | 〜150GB | DB・コンテナイメージ・出力ファイル等を含む |

- **アプリ実行**: すべて **Docker コンテナ**で常駐（**アプリ（Next.js: UI＋API）** / DB / リバースプロキシ）。
- **コンテナ管理（Coolify）**: OSS のセルフホスティング PaaS。**nginx 等のリバースプロキシ + Docker管理 + GitHub連携 + CI/CD** をセットで提供。一度導入すれば細かい設定を意識せず運用でき、その後の管理が楽。
- **接続性**:
  - **社内LANから**: 構築した Ubuntu VM に対し、特別な設定なしで **SSH 接続が可能**。
  - **外部公開（必要な場合）**: **Cloudflare Tunnel** を使い、特定ポートのみ Cloudflare 経由で安全に公開する。VM のポートを直接インターネットに晒さない。
- **ドメイン**: `*.comthink.co.jp` または `*.dev-nct.jp` を社内ですぐ提供可能。本番／開発でサブドメインを割り当てる。

### 3.2 認証・認可

- **認証**: M365（Microsoft Entra ID）による SSO を第一候補。社員アカウントを流用しID管理コストとパスワード管理リスクを削減
  - Entra ID が使えない場合のフォールバックとして手動ID登録（アプリ内認証）
- **認可（権限モデル）**: グループベースのロール管理（RBAC）
  - 一般メンバー（自己評価・他者評価）
  - 評価者（上長・部次長）
  - 人事考課委員会メンバー（閲覧）
  - 管理者（マスタ・項目管理）
- 360°評価は匿名集約のため、**入力者本人を集計結果から逆引きできない**データ設計とする（設計案の「全て匿名」要件に対応）

### 3.3 性能

- 同時利用想定: 対象2部署 × 数十名 → **同時接続10以下**。評価期間（クォーター末等）にアクセスが集中
- 画面応答: 通常操作で 1〜2 秒以内を目標。**コンテナ常駐構成のためコールドスタートは発生せず**、応答が安定する（クラウド・サーバレス案に対する明確な利点）
- 割当スペック（CPU 4C / RAM 16〜32GB）は当該ワークロードに対し十分。バッチ的なスコア集計・帳票出力は数十秒程度を許容

### 3.4 可用性

- 社内利用・期間集中型のため **SLA は緩め**（平日日中の利用を想定、99% 程度で可）
- **オンプレ単一構成のトレードオフ**: VM／物理マシンが単一障害点となる。マルチリージョン等の冗長化は行わない（コスト優先）
- 物理マシン・ホスト Windows Server の保守、停電・ネットワーク断は IDC 運用に準じる。計画停止・障害時はメンテナンス告知＋期日調整で運用回避
- 復旧性は **VM スナップショット + DB バックアップ**で担保（3.7）

### 3.5 拡張性・スケーラビリティ

- まずは VM の割当リソース（CPU 4C / RAM 16〜32GB / ストレージ150GB）の範囲でスケール。**RAM は様子見で増減可能**
- 割当上限に近づいた段階は、まず **VM の割当リソース（RAM・CPU・ストレージ）を増強**して対応する
- **作り直さない**ことを設計上の前提とする（Docker 化により環境を変えず再構築が容易）

#### 3.5.1 スケールの考え方
- スケールはまず **IDC の VM リソース増強**で対応する（ホスト物理マシンの空きリソース範囲内で柔軟に調整）
- **AWS 等クラウドへの移行は現状検討外**。仮に IDC では賄えない規模になった場合の選択肢として、当初検討の記録を **付録A** に残す（必要が生じた段階で改めて評価）

### 3.6 セキュリティ

- 評価情報は**機微情報**として扱う（最小権限・必要範囲のみ閲覧）
- 通信は全て **HTTPS（TLS）** 化。外部公開時は **Cloudflare Tunnel** が TLS を終端し、特定ポートのみを公開
- アクセス制御: 当面は **社内からのアクセスに限定**（社内LANからの SSH／アプリ利用を前提）。外部アクセスが必要な場合のみ Cloudflare Tunnel + 認証（Entra ID）で限定公開
- **監査ログ**: 評価入力・閲覧・エクスポート操作を記録（誰が・いつ・何を）
- シークレット（接続文字列・APIキー）は Coolify の環境変数管理で保持し、コード・Git に含めない
- **脆弱性対応（オンプレの責任分界）**: クラウドと異なり **OS（Ubuntu）・Docker・ミドルのパッチ当ては自社（運用担当）で実施**する。定期的な更新運用を要件化する
- **AI（Claude API）利用時の機微情報の扱い**: 評価情報を外部 API（Anthropic）へ送信する点はオンプレ前提（データを社内IDCに閉じる方針）とのトレードオフ。送信前に**氏名・所属等の個人特定情報をマスキング／匿名化**し、必要最小限のテキストのみを送る。APIキーは Coolify の環境変数で管理。送信可否・範囲・データ保持条件（学習利用の有無等）は契約・運用ポリシーで確定する（要確認事項）

### 3.7 データ・バックアップ

- **履歴保存**: クォーター毎 or 年次で評価履歴を保持（経年の伸びしろ可視化に必要）
- **バックアップ**: ①DB の定期ダンプ ＋ ②Hyper-V の **VM スナップショット** を組合せ、復旧可能性を担保。バックアップ保存先（別ストレージ／別マシン）と頻度を運用で定める
- **データ所在**: 社内IDC（国内・自社管理ラック内）に閉じる。人事情報の所在として明確で、外部クラウドに出さない点はオンプレの利点
- インポート機能は一次案では非対応（手動登録）。エクスポート（PDF/Excel）は機能要件側で対応

### 3.8 運用・監視

- 少人数運用前提。**Coolify による一元管理**でデプロイ・リバースプロキシ・ログ・GitHub連携を集約し、運用負荷を下げる
- デプロイは **GitHub 連携の CI/CD**。`main` → 本番環境、`dev` → 開発環境へそれぞれ自動デプロイ
- ログ・メトリクスは Coolify／コンテナログで確認。必要に応じ軽量な監視を追加
- **OS / Docker / Coolify の更新**は自社運用（クラウドと異なりパッチ責任は自社）。定期メンテナンス枠を設ける

### 3.9 保守性・ポータビリティ

- アプリを Docker 化し、Coolify で管理することで構築・デプロイを定型化。少人数でも運用可能に
- アプリ内で **UI 層と API 層（route handlers）を分離**し、REST 契約を明確化して各層を独立改修可能に
- **ポータビリティ**: Docker 化により特定環境へのロックインを避け、移植性を確保（クラウド移行は現状検討外だが、必要時の余地は残る）

### 3.10 コスト

- **目標**: スモールスタート期のランニングコストを **実質ゼロ**に抑える（既存IDCハードウェアの空きリソースを活用するため）
- **追加で発生しうる費用**は限定的:

  | 項目 | 費用 | 備考 |
  |---|---|---|
  | 物理マシン / ラック / 電気代 | 既存固定費 | 既に稼働中の設備を流用。本件のための新規調達なし |
  | Hyper-V / Windows Server | 既存ライセンス | 稼働中ホストを利用 |
  | Ubuntu VM / Docker / Coolify | ¥0 | すべて OSS・無償 |
  | Cloudflare Tunnel | ¥0 | 無料枠で特定ポート公開が可能 |
  | ドメイン | 既存 | `*.comthink.co.jp` / `*.dev-nct.jp` を流用 |
  | GitHub / CI/CD | 既存 | 既存の GitHub 連携を利用 |

- **クラウド案（付録A）との比較**:

  | 観点 | 社内IDC（採用） | クラウド・サーバレス（付録A） |
  |---|---|---|
  | 追加ランニングコスト | 実質ゼロ（既存設備流用） | 月数千円〜（無料枠次第で変動） |
  | 初期構築の手間 | VM構築 + Coolify導入が必要 | IaC/各サービス設定が必要 |
  | コールドスタート | なし（常駐） | あり（要対策） |
  | 可用性 | 単一構成（トレードオフ） | マネージド標準冗長 |
  | OS/パッチ責任 | 自社 | クラウド側に移譲 |
  | スケール上限 | VM割当の範囲（リソース増強で対応） | オートスケール |
  | データ所在 | 社内IDC（自社管理） | 国内リージョン |

- **方針**: スモールスタートはコスト最優先で **IDC を採用**。全社展開で規模が必要になれば、まず VM リソースを増強して対応する（クラウド移行は現状検討外・3.5.1 / 付録A）

### 3.11 構築・デプロイ方針（WSL検証 → コマンドベース構築 → CI/CD）

> **方針**: 実機（IDCの本番VM）へ直接手作業で構築せず、まず**手元の WSL（Ubuntu）で同等環境を再現して検証**し、**コマンドベース（スクリプト化）で再現可能**にしてから IDC へ適用する。以降の更新は **CI/CD で自動化**する。

- **① ローカル検証（WSL）**
  - 開発者の WSL 上の Ubuntu で、本番VMと同じ **Docker + docker compose + Coolify** 構成をローカル再現し、動作を検証する
  - WSL ↔ IDC本番VM とも **Ubuntu + Docker** で揃えるため、検証した手順がそのまま本番に移植できる（環境差異を最小化）
- **② コマンドベース構築（再現性）**
  - VM 初期セットアップ（Docker / Coolify 導入、必要パッケージ、ファイアウォール、Cloudflare Tunnel 設定等）を**シェルスクリプト等で手順化**し、手作業に依存しない
  - アプリ構成は **`docker compose` / Coolify の定義ファイル**で宣言的に管理し、Git で版管理する
  - 同じスクリプト／定義で、WSL 検証環境 → IDC本番VM → （将来）クラウドを**同一手順で再現**できることを目標とする
- **③ CI/CD（自動デプロイ）**
  - **GitHub** の `main` → 本番、`dev` → 開発 へ、push をトリガに **Coolify / GitHub 連携の CI/CD で自動デプロイ**
  - ビルド（Docker イメージ化）→ デプロイ → リバースプロキシ反映までを Coolify に集約し、手動オペレーションを最小化
- **狙い**: 「手元（WSL）で確認 → コマンドで構築 → 以後は自動デプロイ」を徹底することで、属人化を避け、再構築・移行（IDC→クラウド）を容易にする

## 4. 機能要件

### 4.1 業務フロー
```
[1] 評価項目設定（管理者・部次長）
        ↓
[2] 自己評価入力（被評価者）
        ↓
[3] 360°評価入力（上長・同僚・メンバー）
        ↓
[4] サマリー集計・スコア算出
        ↓
```

## 付録A. クラウド・サーバレス案（当初検討の記録 / 現状は検討外）

> 当初（旧 0.2 / 0.3）はコスト最優先のクラウド・サーバレス前提で検討していた。スモールスタートでは社内IDC（本文3章）を採用するため**非採用**であり、**AWS等クラウドへの移行も現状は検討外**。本付録は当初検討の記録として保持する（将来 IDC で賄えない規模が生じた場合の参考。Docker 化により本文構成からの移行余地は残る）。

### A.1 アーキテクチャ方針（サーバレス）
- マネージドサービスのみで構成し、サーバの構築・運用・パッチ当てを持たない
- フロント: 動的ホスティング（Next.js を SSR）。scale-to-zero 可能なコンテナ実行基盤で動かし、無リクエスト時は課金ゼロ＋CDN を前段に配置
- API: FaaS（関数）。リクエスト時のみ起動し、無リクエスト時は課金ゼロ
- DB: サーバレス／従量課金のマネージドDB（scale-to-zero または最小構成）
- ファイル（PDF/Excel出力等）: オブジェクトストレージ
- 利用が評価期間に集中するワークロード特性に対し、**使った分だけ課金**される構成がクラウドでは最もコスト効率が高い

### A.2 インフラ・ホスティング候補
- 既存IT基盤が **M365（Teams / SharePoint / Entra ID）** 中心のため、認証・運用の親和性が高い **Azure を第一候補**、AWS を対抗候補とする（GCPも可）。

  | 役割 | Azure（第一候補） | AWS（対抗） | 備考 |
  |---|---|---|---|
  | フロント配信（動的/SSR） | Container Apps（従量・scale-to-zero） | App Runner / Amplify Hosting(SSR) | CDN前段・scale-to-zero |
  | API（FaaS） | Functions（従量課金プラン） | Lambda | scale-to-zero |
  | DB | Cosmos DB サーバレス / SQL DB サーバレス | DynamoDB / Aurora Serverless v2 | 従量・自動停止 |
  | 認証 | Entra ID (M365流用) | Cognito + Entra ID 連携 | 3.2 |
  | ストレージ | Blob Storage | S3 | 出力ファイル等 |
  | 監視 | Application Insights | CloudWatch | ログ・メトリクス |

- **CDN/WAF**: クラウド標準のCDN＋WAFを利用（Cloudflare 無料枠の併用も可）

### A.3 ランニングコスト試算（2か月稼働・概算）

> 注: 確定見積りではなく桁感の試算。前提＝Azure 第一候補／国内リージョン（Japan East）／ユーザー〜100名・**同時接続10以下**／評価期間集中・平常時ほぼアイドル／scale-to-zero 構成／WAF・CDN は Cloudflare 無料枠を前段利用。

| サービス | 役割 | 主な無料枠 | 月額目安 |
|---|---|---|---|
| Container Apps | フロント動的ホスティング(SSR) | 月 180k vCPU秒 / 360k GiB秒 / 200万req | ¥0〜500 |
| Functions | API (FaaS) | 月 100万実行 / 40万GB秒 | ¥0 |
| Cosmos DB（サーバレス） | DB | 無料枠適用で 1000RU/s+25GB（1サブスク1個） | ¥0〜1,000 |
| Blob Storage | 出力ファイル等 | — | ¥0〜100 |
| Entra ID | SSO 認証 | 既存 M365 アカウント流用 | ¥0 |
| Application Insights | 監視・ログ | 月 5GB | ¥0 |
| CDN / WAF | 配信・防御 | Cloudflare 無料枠 | ¥0 |
| データ転送（下り） | egress | 月 100GB | ¥0 |

- **想定レンジ（2か月合計）**
  - 無料枠フル活用ケース: **約 ¥0〜2,000**
  - 保守的ケース: **約 ¥4,000〜8,000**（Cosmos 無料枠が使えない／DB のアイドル課金が残る等）
  - → 現実線は **2か月で ¥2,000〜5,000 程度**
- **コストを左右する論点**
  - **DB の選択**: Cosmos DB サーバレス（従量・無料枠）が最安。Azure SQL サーバレスは auto-pause が効かないと最小 vCore 課金で月数千円に膨らむため注意。
  - **WAF/CDN**: Azure Front Door + WAF を使うと月数千円〜。小規模では Cloudflare 無料枠／組込CDN で回避。
  - **常時暖機（コールドスタート対策）**: 入れると scale-to-zero の利点が消えコスト増。小規模では基本不要。

> ※ オンプレ（IDC）採用により、このクラウドコスト自体が不要（追加ランニング実質ゼロ）。クラウド移行は現状検討外であり、本付録はあくまで当初検討の記録・将来の参考として残す。
