## 課題1：上長に聞く内容の整理（＝人事考課の「課題」の整理）

> 議題1に直結。**「何を聞くか」を今週固める**のが前回の宿題。
> 目的＝現行の人事考課の不満・制約を引き出し、システムで解決する対象を絞る。

### 聞くべきこと（案）
- **プロセスと日付**
  - 人事考課委員会はいつ開催か？ その前後のプロセス（自己評価→上長評価→委員会）は？
  - → システムの「期日設計」は人事考課委員会に合わせる必要がある
- **現行の評価項目の課題**
  - 今は「開発／営業」の2分類しかなく、評価項目が書きにくい → ここが一番の不満では？
  - 評価項目を増やしたい（例：資格・技術力・リーダーシップ）が、現状で難しい理由は？
  - クォーターごとの評価は形骸化していないか（「だれている」のでは）
- **評価者・評価範囲**
  - 360°評価で評価者をどこまで広げてよいか（メンバークラスまで入れる？）
  - 上長の評価とメンバーの評価の**比重**はどうあるべきか
  - 最終点数は「課題に対しての達成度」か「メンバーから見た評価」か
  - 現場ごとに人数が違い、評価者の人選が難しい点をどう考えるか
- **委員会に出す目的の確認**
  - 「人事考課委員会に出しやすくする」だけが目的でよいか
  - それとも「伸びしろを見る」など別の価値を持たせたいか（→ これで作る項目が変わる）

## 課題2：他サービス（SaaS）調査・共有

> 議題2に直結。「似たサービスを調べて良いところを取り込む」方針。

### 比較観点（このフレームで各サービスを見る）
| 観点 | 見るポイント |
|---|---|
| 評価の軸 | 組織成果重視か／フラット・汎用評価か |
| 評価方式 | 絶対評価／相対評価、選択式（性格診断的）か記述式か |
| 評価者範囲 | 上長のみ／360°／メンバー含む |
| 項目設定 | 項目をどこまで自由に作れるか |
| 料金 | 無料枠・小規模で使えるか |

### 調べる対象（案）
- **SmartHR**：部・課の成果重視。差別化の比較対象として必須
- **360°評価系SaaS**（例：あしたのチーム、HRBrain、カオナビ 等）
- **診断系**：MBTI的な選択式診断 → 「答えるのが楽」というアイデアと相性

### 差別化メモ
- SmartHRとの違い＝「部の評価軸からの脱却」「組織に依存しないフラットな評価」
- 選択式（性格診断的）にすると回答負荷が低くて良いかも、という前回の意見を検証する

## 課題3：機能要件の整理（各自まとめて持ち寄り）

> 議題3に直結。野原・本荘・幡野で分担。各自の案を持ち寄って議論。

### コア機能フロー（前回の合意イメージ）
```
自己評価（ステータス管理）→ 360°評価 → サマリー →（余裕あればダッシュボード）
```

### 機能候補の整理
- **自己評価**：ステータス管理（提出済／未提出 など）
- **360°評価**：評価者の割り当て、入力フォーム
- **評価項目設定**：部門長／部次長が項目を捻出できる仕組み（協力が前提）
  - 候補項目：資格・技術力・リーダーシップ
  - MBTI的な選択式の診断項目も検討
- **サマリー**：個人ごとの集計
- **ダッシュボード**：余裕があれば（優先度低）
- **権限管理**：グループ式の権限設定（自己／上長／管理者 など）

### 詰めるべき論点（会議で出たもの）
- 最終点数の出し方（課題達成度 vs メンバー評価、人数差の補正）
- 上長評価とメンバー評価の比重
- 評価者の範囲（メンバークラスまで含めるか）

## 課題4：非機能要件・インフラ（野原・折登）

> 議題には明示されていないが、自分の担当として並行整理しておく。

### 課題1：インフラ基盤をどうするか（「箱」検討）
- **社内のみ公開** か **外部にも公開** か
  - → 一旦「**社内のみ可**」に倒し、外部公開は課題として棚上げ（コスト問題を回避できる）
- 候補
  - 社内（M365 など）：社内のみなら用意可能
  - 外部無料枠：GCP / Xserver / Oracle 無料枠
  - さくらサーバー：微妙か
  - AWS：移行が容易（Gitに上げて移行しやすくしておく）／Cloudflare 併用案
- **コスト負担**：誰が払うのか → 社内のみなら回避。外部は要相談

### その他
- **ID登録**：手動登録で開始（インポートデータはありそうだが、盛り込むと面倒 → 後回し）
- スモールスタート前提：まずは評価フローを考える程度でOK。案が良ければ仕組みが変わって評価されるはず
