# エラーメッセージ規約

reader-eval（Next.js フルスタック）における、ユーザー向け日本語エラーメッセージの文言ルールを定める。レスポンス形式・`code` 語彙・例外変換方針は [`api-response.md`](api-response.md) を正とし、本ファイルは **文言（言い回し）**に特化する。

新規・既存メッセージを追加 / 変更する際は §3 のパターンに従い、§4 のチェックリストで確認すること。

## 1. 適用範囲

- route handler / server actions が返す `ErrorResponse.message` および `errors` の各値
- UI（Server / Client Components）で表示する toast・アラート・フォームのフィールド別メッセージ・確認ダイアログの文言
- ユーザーの目に触れる文言はすべて **日本語**。技術用語（内部例外名・stack trace・`null` 等）を含めない（§3.4）。

## 2. 基本原則

- 文言は **エンドユーザーが読んで対処できる**ことを最優先する。
- 同じ意味のメッセージで言い回しを揺らさない（規約形に統一）。
- 入力バリデーションは可能な限り **フィールド別**（`errors` マップ）に出し、フォーム項目の近くで表示する。

## 3. 文言ルール

### 3.1 文末（用途別パターン）

| カテゴリ | 文末 | 例 |
| --- | --- | --- |
| 必須項目 | 「は必須です」 | 「メールアドレスは必須です」 |
| 形式エラー | 「の形式が不正です」 | 「メールアドレスの形式が不正です」 |
| 文字数上限 | 「は{max}文字以内で入力してください」 | 「氏名は100文字以内で入力してください」 |
| 文字数範囲 | 「は{min}〜{max}文字で入力してください」 | 「コメントは10〜2000文字で入力してください」 |
| 権限エラー | 「権限がありません」 | 「この操作を実行する権限がありません」 |
| 状態遷移エラー | 「には{action}できません」 | 「確定済みの評価には編集できません」 |
| 期限切れ | 「の有効期限が切れています」 | 「評価期間の有効期限が切れています」 |
| 外部連携失敗 | 「に失敗しました」 | 「AI による評価の生成に失敗しました」 |
| リソース未存在 | 「が見つかりません」 | 「参加者が見つかりません」 |
| 成功メッセージ | 「を{action}しました」 | 「評価を保存しました」 |
| 汎用エラー | 「エラーが発生しました」 | 「ネットワークエラーが発生しました」 |

「以内で入力してください」と「以内です」のような短縮形の混在は **避ける**（規約形は「入力してください」）。

### 3.2 主語・用語の統一

- メール: 「メールアドレス」で統一（「Email」「メアド」は使わない）
- ユーザー: 「ユーザー名」（人物の表示名は「氏名」）
- 研修対象者: 「参加者」（「受講者」「対象者」と揺らさない）
- 評価: 「評価」（「採点」「スコア」は文脈上必要な場合のみ）
- 用語の揺れは本ファイルと CLAUDE.md の用語を正とする。新語が必要なら本ファイルに追記する。

### 3.3 削除・確認ダイアログ（UI）

統一形: **「『{対象名}』を{action}しますか？」**（対象名は二重カギで強調）。

- OK 例: 「『2026年度上期 評価』を削除しますか？」
- NG 例: 「この評価を削除してよろしいですか？」「本当に削除しますか？」「削除してもよろしいですか」

### 3.4 技術用語の混入禁止

`null` / `undefined` / stack trace / 内部例外名 / SQL・テーブル名 / HTTP ステータス数値などの技術詞彙を **ユーザー向けメッセージに含めない**。技術詳細は **サーバログ**（`console.error` 等）にのみ出し、ユーザーには §3.1 の規約形で表示する。

特に `INTERNAL_ERROR`（500）は固定文言「サーバーエラーが発生しました」を返す（[`api-response.md`](api-response.md) §4）。

### 3.5 AI（Claude）連携時のメッセージ

- AI 呼び出しの失敗は「AI による{処理}に失敗しました」の形（例: 「AI による評価の生成に失敗しました」）。
- API キー・モデル名・レート制限の詳細など内部事情はユーザーへ出さず、ログに記録する。
- タイムアウト / 一時的失敗で再試行可能な場合は「時間をおいて再度お試しください」を添える。

## 4. 新規 / 既存メッセージを追加・変更する際のチェックリスト

メッセージを追加 / 変更する **前** に確認する。1 つでも No なら手を止めて本ファイルと [`api-response.md`](api-response.md) を確認する。

- [ ] §3.1 の文末・§3.2 の主語 / 用語・数値表現の規約に合致している
- [ ] 該当する `code`（[`api-response.md`](api-response.md) §2.1）と HTTP ステータスを正しく選んでいる
- [ ] 入力エラーは可能なら `errors` マップでフィールド別に出している
- [ ] 削除 / 確認ダイアログは §3.3 の統一形に従っている
- [ ] 技術用語（`null` / 内部例外名 / stack trace / SQL 等）が混入していない
- [ ] `INTERNAL_ERROR` は固定文言で、技術詳細はログのみに出している
- [ ] 同種のメッセージで既存の言い回しと揺れていない
