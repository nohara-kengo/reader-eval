# 認可・RBAC 規約

reader-eval（リーダー研修 評価システム）における **認可（誰が何をできるか）** の方針を定める。
**認証（誰であるか）** は Entra ID（SSO）で行い（[`app.md`](app.md) §認証）、本ファイルは認証済みユーザーに対する **権限制御（RBAC）** を正とする。

> **ロール / 権限マトリクスは暫定（要件確定で見直す）。** 業務要件に合わせて値を更新する。「暫定」と明記した箇所は確定後に追記・修正する。

## 1. ロール定義（暫定）

| ロール | 識別子（`as const`） | 概要 | 主な操作 |
| --- | --- | --- | --- |
| 一般メンバー | `MEMBER` | 研修参加者・評価入力者 | 自己評価の入力 / 他者評価（360°）の入力 / 自分に関する確定結果の閲覧 |
| 評価者（上長・部次長） | `EVALUATOR` | 部下・配下メンバーの評価者 | 担当範囲のメンバー評価入力 / 担当範囲の評価結果の閲覧 |
| 人事考課委員会 | `HR_COMMITTEE` | 考課判断のための閲覧者 | 全体の評価結果・集計の **閲覧のみ**（入力・編集不可） |
| 管理者 | `ADMIN` | システム / マスタ管理 | 評価項目・期間・対象者などのマスタ管理 / ロール割当 / 全体運用 |

- ロール定数は **文字列ユニオン / `as const`** で `lib/shared/`（共通コード値）に定義する（[`shared.md`](shared.md) §3 / [`naming-conventions.md`](naming-conventions.md) §3）。enum は使わない。
- ロールは **複数保持し得る**（例: 評価者かつ一般メンバー）。判定は「いずれかのロールが権限を満たすか」で行う。

```ts
// lib/shared/roles.ts（雛形・暫定）
export const ROLES = {
  MEMBER: "MEMBER",
  EVALUATOR: "EVALUATOR",
  HR_COMMITTEE: "HR_COMMITTEE",
  ADMIN: "ADMIN",
} as const;
export type Role = (typeof ROLES)[keyof typeof ROLES];
```

## 2. 権限マトリクス（暫定）

操作カテゴリ別の許可（◯=許可 / △=条件付き / ×=不可）。**暫定** であり要件確定で更新する。

| 操作 | MEMBER | EVALUATOR | HR_COMMITTEE | ADMIN |
| --- | --- | --- | --- | --- |
| 自己評価の入力 / 更新 | ◯（自分のみ） | ◯（自分のみ） | × | × |
| 他者評価（360°）の入力 | ◯（割当対象のみ） | ◯（担当範囲） | × | × |
| 自分の確定結果の閲覧 | ◯（自分のみ） | ◯（自分のみ） | — | — |
| 担当メンバーの評価結果の閲覧 | × | △（担当範囲のみ） | × | × |
| 全体の評価結果・集計の閲覧 | × | × | ◯（閲覧のみ） | ◯ |
| 評価項目 / 期間 / 対象者マスタの管理 | × | × | × | ◯ |
| ロール割当・権限管理 | × | × | × | ◯ |
| 監査ログの閲覧 | × | × | × | ◯ |

- △（条件付き）は **データ単位のスコープ判定** が必要（例: 評価者は「自分の担当範囲のメンバー」のみ）。ロール一致だけでなく **対象データの所有 / 担当関係** をサービス層で必ず検証する。
- 「閲覧のみ」ロール（`HR_COMMITTEE`）には **書き込み系の操作を一切露出しない**（UI・API 両方）。

## 3. Entra ID グループ → アプリロールのマッピング

- 認証は Entra ID（SSO）。**Entra ID のセキュリティグループを、アプリ内のロールへマッピング**して認可に用いる。
- マッピングは **`lib/auth/` に集約**する（マッピング表・変換関数を 1 箇所に置く）。ロール判定ロジックを各機能に散らさない。
- グループ ID → ロールの対応表は **環境変数 / 設定** で外出しする（テナント・環境ごとに異なるため。シークレット類は [`security.md`](security.md) に従う）。グループ ID をコードへハードコードしない。
- ユーザーが **どのグループにも該当しない場合は権限なし**（既定拒否）。暗黙の昇格をしない。
- トークン / グループ情報の検証は **サーバ側でのみ** 行う（[`app.md`](app.md) §認証）。

```ts
// lib/auth/roles.ts（方針イメージ・暫定）
// Entra ID グループ ID → アプリロール（対応は環境変数で注入）
export function mapGroupsToRoles(groupIds: string[]): Role[] {
  // 設定（環境変数）の対応表に基づき変換。該当なしは空配列（＝権限なし）
}
```

## 4. 認可ガード（route handler / server action）

- **すべての route handler / server action の入口で認可を必須にする**（[`app.md`](app.md) §認証 / [`service-layer.md`](service-layer.md) §1）。「公開エンドポイント」を作る場合は明示的に許可する（既定は要認証・要認可）。
- 判定順序: **認証 → 認可 → 入力検証 → サービス呼び出し**。
  - セッション無効 / 失効 → **401 `UNAUTHORIZED`**
  - 認証済みだが権限不足 → **403 `FORBIDDEN`**
  - レスポンス形式・`code` 語彙は [`api-response.md`](api-response.md)、例外型（`UnauthorizedError` / `ForbiddenError`）は [`error-handling.md`](error-handling.md) を正とする。
- 認可エラーで **内部実装（どのチェックで弾いたか・対象の存在有無）を過度に露出しない**（[`api-response.md`](api-response.md) §4）。存在するが権限がないリソースは、必要に応じて 404 と同等の見せ方にして存在を秘匿する判断もする。

```ts
// app/api/.../route.ts（イメージ）
export const POST = withErrorHandler(async (req) => {
  const session = await requireSession();          // 無効なら UnauthorizedError(401)
  requireRole(session, [ROLES.EVALUATOR, ROLES.ADMIN]); // 権限不足なら ForbiddenError(403)
  const input = parse(SchemaZod, await req.json());
  const result = await someService(db, session, input); // データ単位のスコープ判定はサービス層
  return Response.json(result, { status: 201 });
});
```

- **サーバ側の判定を正**とする。UI の出し分け（ボタン非表示・メニュー制御）は **UX 補助に過ぎず、認可の代替にしない**。クライアントの判定を信頼しない。
- データ単位のスコープ（自分のもの / 担当範囲か）の検証は **サービス層** で行う（[`service-layer.md`](service-layer.md) §2）。ロール一致だけで通さない。

## 5. 最小権限の原則

- 既定は **拒否**。必要な操作にのみ明示的に権限を与える。
- ロールには **業務上必要な最小限の操作のみ** を割り当てる。「とりあえず広めに許可」をしない。
- 「閲覧のみ」のロールに書き込み経路を残さない。新規エンドポイント追加時は、対象ロールを §2 のマトリクスで再確認する。
- 管理者操作（マスタ変更・ロール割当）は **監査ログ** に残す（[`error-handling.md`](error-handling.md) §5）。

## 6. 360°評価の匿名性（要件対応）

360°評価では、**入力者を集計結果から逆引きできないデータ設計**を必須とする。

- **匿名性の保証はデータ設計とアクセス制御で担保する**（UI で隠すだけにしない）。
- 集計・閲覧用の結果からは、**個々の入力者（誰がどの点を付けたか）を特定できないようにする**。
  - 集計表示には **入力者識別子を含めない**。閲覧 API / DTO に評価者の個人特定情報を載せない。
  - 少人数の回答グループでは **個別回答が事実上特定され得る**ため、最小回答件数の閾値（暫定: 担当群が一定数未満なら個別表示・集計開示を制限）を設ける。閾値は要件確定で決める（暫定）。
- 入力の保存と集計の参照を分離し、**集計経路から評価者 ID をたどれない**設計にする（例: 集計用ビュー / 集計テーブルには評価者識別子を持たせない、または非可逆化する）。具体的なスキーマは ORM 選定（ADR）後に確定する（暫定）。
- 匿名性に関わる例外（自己評価・上長評価など、入力者が明示される種別）は **評価種別ごとに匿名性ポリシーを分けて** 定義する（暫定: 360°他者評価＝匿名 / 自己評価・上長評価＝記名）。
- 監査ログには操作の事実を残すが、**匿名性を破る形で集計と評価者を紐付けて閲覧できる権限を一般運用に与えない**（管理者であっても匿名性ポリシーを尊重する）。

## 7. 禁止事項

- クライアント側判定のみで認可を済ませる（UI 出し分けを認可の代替にする）。
- route handler / server action で認可ガードを省略する（既定は要認可）。
- Entra ID グループ ID / ロール対応表のハードコード、ロール判定ロジックの各機能への散在。
- 「閲覧のみ」ロールへの書き込み経路の露出。
- 360°評価の集計結果から入力者を逆引きできるデータ設計 / API レスポンス。
