# Claude / AI 利用規約

reader-eval（リーダー研修 評価システム）における **Claude（Anthropic API、`@anthropic-ai/sdk`）** の利用方針を定める。AI は評価業務の **支援** であり、評価そのものの責任を代替しない（最終判断は人が行う）。

本ファイルは AI 利用の方針を扱う。アプリ全体の構成は [`app.md`](app.md)、外部サービス連携の実装パターンは [`service-layer.md`](service-layer.md)、エラー処理は [`error-handling.md`](error-handling.md) を正とする。

> **機微情報のマスキングは [`security.md`](security.md) を正**とする。本ファイルは「送信前に必ず適用する」ことのみを述べ、マスキング規則の定義は重複させない（§5）。

## 1. モデル選定

- **既定モデルは `claude-opus-4-8`**（環境変数 `ANTHROPIC_MODEL` の既定値）。
- 用途別の選定基準（**暫定**。実コスト・品質を計測してから [`config.md`](config.md) と整合させ確定する）:

| 用途 | モデル（暫定） | 理由 |
| --- | --- | --- |
| 高度な要約 / 分析（伸びしろ分析・委員会向けドラフト等） | `claude-opus-4-8`（既定） | 推論品質を優先 |
| 定型・大量処理（多数の評価コメントの整形・分類） | `claude-sonnet-4-6` | 品質とコストのバランス |
| 軽量・低コスト（短文整形・簡易チェック） | `claude-haiku-4-5` | 低コスト・低レイテンシ |

- モデル ID は **コードに直書きしない**。`lib/claude/` の設定（環境変数 / 定数）から解決する。
- モデルを切り替える場合は理由を PR に記し、選定基準表を更新する。

## 2. 呼び出しの集約（サーバ側のみ）

- Anthropic クライアントの生成・呼び出しは **`lib/claude/` に集約**する。features / route handler / コンポーネントから SDK を直接 import しない。
- **サーバ側からのみ**呼ぶ。**API キー（`ANTHROPIC_API_KEY`）をクライアントへ露出しない**（[`app.md`](app.md) 禁止事項）。
- AI 機能はサービス層から `lib/claude/` のクライアント経由で利用する（[`service-layer.md`](service-layer.md) §4）。route handler / server action に呼び出しロジックを直書きしない。
- 呼び出しには **タイムアウトを必須**とする（§7）。

```
features/<feature>/server/  → lib/claude/（クライアント / プロンプト / 呼び出し）→ Anthropic API
```

## 3. プロンプト管理

- プロンプトは **`lib/claude/prompts/` に集約しコードとして管理**する。文字列をサービス内に散在させない。
- プロンプトは **バージョン管理・PR レビューの対象**とする（評価の公平性に直結するため、変更は履歴に残す）。
- プロンプトには**識別子 / バージョン**を持たせ、出力ログ（[`logging.md`](logging.md)）から「どのプロンプトで生成したか」を追跡できるようにする。
- 動的な値（評価データ等）はテンプレートの引数として渡し、文字列連結で組み立てない（インジェクション・表記ゆれ防止）。
- 評価対象者由来のテキストを **指示として解釈させない**（プロンプトインジェクション対策。データはデータ領域に明示して渡す）。

## 4. トークン / コスト管理

- プロンプトに渡すテキストは **必要最小限**にする。評価データ全件ではなく、当該処理に必要な範囲のみを渡す。
- 呼び出しごとに **`max_tokens` の上限**を設定する（用途別に妥当な値を `lib/claude/` で管理。暫定値は実測で調整）。
- 利用量（モデル / 入出力トークン / 用途 / 相関 ID）を **ログに記録**する（[`logging.md`](logging.md)）。コスト把握と異常検知に用いる。
- 同一入力の繰り返し呼び出しを避ける（必要に応じて結果を保存・再利用。トランザクション内では呼ばない＝[`service-layer.md`](service-layer.md) §3）。

## 5. 機微情報の取り扱い

- **送信前に必ずマスキングを適用**する。評価対象者の個人情報・社内機微情報を素のまま Anthropic API に送らない。
- **マスキング規則の定義は [`security.md`](security.md) を正**とし、本ファイルでは重複定義しない。
- マスキングは `lib/claude/` の送信経路で **確実に通る**設計とし、個別呼び出し側の実装に依存させない。

## 6. 出力の扱い

- 構造化された出力が必要な場合は、**出力をスキーマ検証**する（Zod 等）。検証失敗はドメイン外エラーとして扱い、画面の評価機能本体を壊さない。
- AI 出力は **ハルシネーション前提**で扱う。評価コメント・サマリー・ドラフトは **人手レビューを残す**（AI 出力を確定値として無検査で保存しない）。
- 生成物は「AI による下書き / 補助」であることが UI 上分かるようにする（人が最終確定する運用を崩さない）。
- 出力を画面に出す際の文言・エラー表示は [`error-message.md`](error-message.md) を正とする。

## 7. 失敗時の方針（タイムアウト / リトライ / フォールバック）

- **タイムアウト必須**・リトライ（指数バックオフ・回数上限）は [`service-layer.md`](service-layer.md) §4 に従う。
- DB トランザクション内で AI を呼ばない（外部 I/O 禁止。コミット後に実行する＝[`service-layer.md`](service-layer.md) §3）。
- **評価機能本体を止めないフォールバック**を採る:
  - AI 支援が失敗しても、評価の入力・保存・閲覧は継続できること。
  - 当該 AI 支援機能のみ無効化する / 後で再実行できる導線を用意する。
- ユーザー向け文言は「AI による{処理}に失敗しました」（[`error-message.md`](error-message.md) §3.5）。API キー・モデル名・レート制限等の内部事情は出さず、ログに記録する（[`error-handling.md`](error-handling.md) §6）。

## 8. 想定用途（reader-eval）

いずれも **AI は支援、確定は人** を前提とする:

- 評価コメントの要約 / 整形
- 自己評価・360°評価のサマリー生成
- 評価委員会向けドラフトの作成
- 伸びしろ（強み・改善点）の分析

新たな AI 用途を追加する際は、§5 のマスキング・§6 の人手レビュー・§7 のフォールバックを満たすことを確認する。
